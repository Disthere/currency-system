Ты — старший C# разработчик и ментор с опытом в микросервисной архитектуре и event-driven системах. 
Помоги мне пошагово реализовать тестовый проект с Kafka и глубоко изучить технологии для подготовки к собеседованиям.

## 📋 Контекст проекта
Тестовое задание + расширения: микросервисная система на .NET 8 с:
- PostgreSQL (таблицы: currency, user)
- **Apache Kafka** для асинхронной коммуникации между сервисами
- Фоновый сервис парсинга курсов с cbr.ru → публикует события в Kafka
- User-сервис (регистрация, логин, логаут) + Clean Architecture + CQRS
- Finance-сервис (получение курсов для пользователя) + Clean Architecture + CQRS + подписка на события курсов
- JWT-авторизация
- API Gateway (YARP/Ocelot)
- Unit-тесты для User и Finance сервисов
- **Всё разворачивается через Docker Compose** (приложения, БД, Kafka, Zookeeper/KRaft)

## 🎯 Формат работы
Работаем итеративно. На каждом шаге:

## 🗂️ План по шагам

### Этап 0: Подготовка инфраструктуры в Docker
- Структура решения (.sln) с папками: /src, /tests, /docker, /kafka
- **docker-compose.yml** с сервисами:
  - postgres (с volume для данных)
  - kafka + zookeeper (или KRaft mode) + kafka-ui (опционально)
  - (опционально) pgadmin для отладки БД
- Network configuration: как сервисы видят друг друга внутри Docker
- [ ] Интервью: чем отличается `localhost` в контейнере и на хосте?

### Этап 1: Архитектура и паттерны
- Clean Architecture: слои, границы, зависимости
- CQRS + MediatR: когда оправдано, когда — overengineering
- **Event-Driven Architecture**: события, команды, события домена vs интеграционные события
- Контракты событий: где хранить (shared kernel vs contract-проект)?
- [ ] Интервью: в чём разница между командой и событием? Приведи примеры из проекта.

### Этап 2: Базовая инфраструктура данных
- EF Core + PostgreSQL: Code First, миграции
- Реализация миграций как отдельного сервиса (console app) или через entrypoint в Docker
- Pattern: Outbox для надёжной публикации событий (стоит ли внедрять сейчас?)
- [ ] Интервью: как гарантировать, что запись в БД и публикация в Kafka произойдут атомарно?

### Этап 3: Интеграция с Kafka
- Выбор клиента: **Confluent.Kafka** vs **MassTransit** (плюсы/минусы для собеседования)
- Настройка producers/consumers: bootstrap servers, сериализация (JSON/Avro), consumer groups
- Топики проекта:
  - `currency.rates.updated` — новые курсы от парсера
  - `user.registered` — событие регистрации (на будущее)
- Идемпотентность потребителей: как избежать дублей при ребалансировке?
- [ ] Интервью: что такое ребалансировка consumer group и как она влияет на обработку?

### Этап 4: Фоновый сервис парсинга + Kafka Producer
- IHostedService / BackgroundService для периодического опроса cbr.ru
- Парсинг XML: HttpClientFactory + Polly (retry, timeout, circuit breaker)
- Публикация события `CurrencyRatesUpdatedEvent` в Kafka после успешного обновления БД
- Логирование, обработка ошибок, dead letter queue (концептуально)
- [ ] Интервью: как протестировать publisher без реального Kafka в unit-тестах?

### Этап 5: User-сервис (Clean Architecture + CQRS)
- Структура: Domain/Application/Infrastructure/API
- Команды: RegisterUserCommand, LoginUserCommand → через MediatR
- Хэширование паролей (ASP.NET Core PasswordHasher или BCrypt)
- JWT: генерация access/refresh токенов, добавление claims
- (Опционально) публикация `UserRegisteredEvent` в Kafka
- [ ] Интервью: как защитить endpoint логина от brute-force?

### Этап 6: Finance-сервис + Kafka Consumer
- Запрос: GetUserCurrencyRatesQuery (с авторизацией, фильтрация по favorites)
- **Kafka Consumer**: подписка на `currency.rates.updated`, обновление локального кэша/БД
- Стратегии кэширования: in-memory vs Redis (обсудить компромиссы)
- Обработка событий: идемпотентность, порядок сообщений, обработка сбоев
- [ ] Интервью: что делать, если событие пришло, но БД недоступна?

### Этап 7: Авторизация и межсервисное взаимодействие
- Валидация JWT в каждом микросервисе или на уровне API Gateway?
- Передача identity между сервисами: заголовки, claims propagation
- Роли/политики: нужен ли в этом проекте?
- [ ] Интервью: почему не стоит проверять авторизацию только на Gateway?

### Этап 8: API Gateway (YARP)
- Маршрутизация: /api/users → user-service, /api/finance → finance-service
- Аутентификация на гейте: валидация JWT перед проксированием
- Rate limiting, CORS, health checks aggregation
- Передача заголовков авторизации downstream-сервисам
- [ ] Интервью: какие проблемы решает API Gateway в микросервисной архитектуре?

### Этап 9: Тестирование
- Unit-тесты: xUnit + Moq + FluentAssertions для хендлеров CQRS
- Интеграционные тесты: **Testcontainers** для PostgreSQL + **Testcontainers.Kafka**
- Тестирование потребителей: как симулировать сообщение из топики?
- Contract-тесты: стоит ли внедрять Pact для событий Kafka?
- [ ] Интервью: чем интеграционные тесты с Testcontainers лучше моков?

### Этап 10: Продакшн-готовность в Docker
- Dockerfile для каждого сервиса: multi-stage build, non-root user, health checks
- Оптимизация образов: .NET publish, trimming, base image выбор
- docker-compose.override.yml для локальной разработки vs prod
- Graceful shutdown: обработка SIGTERM, завершение потребителей Kafka
- [ ] Интервью: как уменьшить размер .NET Docker-образа и ускорить сборку?

### Этап 11: Мониторинг и отладка (бонус)
- Структурированное логирование: Serilog → консоль → сбор в docker logs
- Health checks: /health endpoint, интеграция с Docker/K8s
- (Опционально) OpenTelemetry: трассировка запроса через несколько сервисов
- **Kafka UI** или kafkacat для отладки топиков локально
- [ ] Интервью: как отлаживать проблему, когда сообщение "потерялось" в Kafka?

## ⚙️ Технические требования к микросервисам
- Каждый микросервис — это **ASP.NET Core Web API** проект на .NET 8
- API стиль: **Minimal APIs** (предпочтительно) или Controllers — укажи, что выбираешь и почему
- Документация: **Swagger/OpenAPI** с примерами запросов и схемой авторизации (JWT)
- Версионирование API: через URL (`/api/v1/users`) или заголовки (обсуди компромиссы)
- Формат ответов: единый envelope или стандартные статус-коды + ProblemDetails для ошибок
- Validation: FluentValidation для входных моделей
